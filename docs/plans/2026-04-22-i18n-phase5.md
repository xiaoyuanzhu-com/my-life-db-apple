# Phase 5 — Apple Native i18n (en + zh-Hans)

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Localize the MyLifeDB Apple native app and Share Extension into English and Simplified Chinese (zh-Hans), using Xcode 15+ String Catalogs (`.xcstrings`). System locale drives display — no in-app language picker in this phase (deferred).

**Architecture:**
- **String Catalog** — One `Localizable.xcstrings` per target (main app + share extension). New-format JSON; Xcode auto-populates source strings from `String(localized:)` / `LocalizedStringKey` / `Text("...")` usages at build time.
- **Info.plist permissions** — `InfoPlist.xcstrings` per target holds the 4 NS*UsageDescription strings.
- **No runtime locale override** — `Locale.current` / `Bundle.main.preferredLocalizations` are authoritative; user changes language via iOS/macOS System Settings. (In-app picker is a Phase 6 concern.)
- **SwiftUI automatically treats `Text("literal")` as a `LocalizedStringKey`.** Most strings already benefit without code changes — they just need to be added to the String Catalog with a zh-Hans translation. The real work is (a) creating the catalog, (b) migrating strings used in non-LocalizedStringKey contexts (navigation titles sometimes, Button action labels passed as String, .alert titles, accessibility labels), (c) localizing struct fields (DataSource.name etc.), and (d) translating.

**Tech Stack:** Xcode 15+ String Catalogs, SwiftUI, Swift 6, `String(localized:)`, `LocalizedStringResource`, `Text(_:tableName:)`, `RelativeDateTimeFormatter`.

---

## Scope decisions (locked)

- **Targets:** Main app (`MyLifeDB`) + Share Extension (`MyLifeDBShareExtension`). Tests NOT in scope.
- **Locales shipped:** `en` (source) + `zh-Hans`.
- **String count:** ~150–170 unique user-visible strings across both targets.
- **NO in-app language picker.** System-driven only.
- **NOT localizing `DataSource.description` fields** (not rendered in UI — survey confirmed).
- **NOT localizing** unit-test strings, debug log messages, or preview sample data.
- **Verification:** `xcodebuild` on macmini for iOS Simulator + macOS (build only; no simulator UI smoke).

### Step 2.1: `AppTab` — separate rawValue from UI label

In `MainTabView.swift` lines ~19-31, the tab labels come from `AppTab.rawValue` ("Data", "Agent", "Explore", "Me"). Add a computed property returning a `LocalizedStringResource`:

```swift
enum AppTab: String, Hashable {
    case data = "Data"
    case agent = "Agent"
    case explore = "Explore"
    case me = "Me"

    var label: LocalizedStringResource {
        switch self {
        case .data: "Data"
        case .agent: "Agent"
        case .explore: "Explore"
        case .me: "Me"
        }
    }
}
```

Then at each use site (e.g. `Tab(AppTab.data.rawValue, ...)` and `Label(tab.rawValue, ...)`), replace `.rawValue` with `.label`. `Tab` and `Label` accept `LocalizedStringResource`-bearing initializers.

### Step 2.2: `InboxTimestampView` — replace hand-rolled relative time

In `InboxTimestampView.swift` lines 27-49, delete the entire `timeAgoString(from:)` function body with English string literals. Replace with:

```swift
private static let relativeFormatter: RelativeDateTimeFormatter = {
    let f = RelativeDateTimeFormatter()
    f.unitsStyle = .full
    f.dateTimeStyle = .named  // gives "yesterday", "today" instead of "1 day ago"
    return f
}()

private func timeAgoString(from date: Date) -> String {
    let elapsed = Date().timeIntervalSince(date)
    if elapsed < 60 {
        return String(localized: "Just now")
    }
    return Self.relativeFormatter.localizedString(for: date, relativeTo: Date())
}
```

`RelativeDateTimeFormatter` automatically handles plural/singular in every locale. Delete the `DateFormatter()` fallback from line 46-49 unless the absolute format is used elsewhere — grep `timeAgoString` and related helpers.

### Step 2.3: `SyncCalendarGrid` weekday headers

In `SyncCalendarGrid.swift` around line 71, replace the hardcoded array:
```swift
let weekdayLabels: [(Int, String)] = [(0,"M"),(1,"T"),(2,"W"),(3,"T"),(4,"F"),(5,"S"),(6,"S")]
```

With:
```swift
let cal = Calendar.current
// reorder from Sunday-first to Monday-first
let symbols = cal.veryShortStandaloneWeekdaySymbols
let rotated = Array(symbols[1...]) + [symbols[0]]  // Mon..Sat, Sun
let weekdayLabels = rotated.enumerated().map { ($0.offset, $0.element) }
```

Also at line 123, for `shortMonthSymbols`, use `Calendar.current.shortMonthSymbols` (not `Calendar(identifier: .gregorian).shortMonthSymbols` — the latter doesn't carry the user's locale).

### Step 2.4: `AgentSessionListView.shortRelativeTime` and section labels

Lines ~552-560 hand-rolled `shortRelativeTime()` returns `"5s"`, `"3m"`, etc. Keep as-is (universal numeric abbreviations ARE locale-appropriate in many locales including Chinese — `5秒` / `3分` has more visual clash than keeping `5s`/`3m`). Annotate with a comment so future localizers know it's intentional:

```swift
// Note: compact numeric abbreviations (5s, 3m, 2h, 4d) are intentionally
// kept locale-independent for readability in a cramped list cell.
```

But wrap the time-section labels (`"Today"`, `"Yesterday"`, `"Past Week"`, `"Past Month"`, `"Earlier"`, `"Sessions (All)"`, etc.) as `String(localized: ...)` where they're consumed as `String`. If they're fed into `Text(...)` directly they're already `LocalizedStringKey`.

Also the `.accessibilityLabel("Agent is working")` / `"Unread results"` → `.accessibilityLabel(String(localized: "Agent is working"))`.

## Task 3 — Migrate main target string literals

Every user-visible string literal needs to flow through String Catalog. In SwiftUI, `Text("foo")` and `Button("foo") { }` already take `LocalizedStringKey` — no code change needed. The real migration work is for strings passed as `String`:

1. `.navigationTitle(someStringVariable)` when `someStringVariable` is a `String`, not a literal → use `String(localized: "...")`.
2. `.alert("Title", ...)` → already `LocalizedStringKey`, fine.
3. `.accessibilityLabel("...")` → already `LocalizedStringKey` if literal; if passed a `String` variable → wrap with `String(localized:)`.
4. Error messages in `APIError.swift` `errorDescription` + `userMessage` — these are `String`-typed, wrap the literals with `String(localized:)`.
5. `ContentUnavailableView("Title", ...)` — `LocalizedStringKey`, fine.
6. `enum Platform: String { case .iPhone = "iPhone" }` where `rawValue` is rendered — same pattern as Task 2.1: add a `displayName: LocalizedStringResource`.
7. `DataCategory.name`, `DataSource.name` fields (String) — change type to `LocalizedStringResource` OR keep as String and wrap at render site with `String(localized: String.LocalizationValue(source.name))`. Prefer the first approach: change the struct fields to `LocalizedStringResource`.

### Step 3.1: APIError.swift

All `errorDescription: String?` and `userMessage: String` literal returns → wrap with `String(localized:)`. Example:
```swift
case .invalidURL: return String(localized: "Invalid URL")
```

### Step 3.2: DataCollectView.swift — the big file

struct DataCategory: Identifiable {

enum Platform: String {  // keep rawValue for identity/persistence if needed

NOTE: `LocalizedStringResource` literals in Swift 5.9+ accept a string literal directly (the magic `ExpressibleByStringLiteral` conformance makes the source strings extractable at build time).

All ~80 `DataSource(name: "Daily step count", ...)` entries in the `dataCategories` array need no syntax change — the literal string is compatible.

Render sites in `DataSourceRow`:
- `Text(source.name)` → already works (`Text` accepts `LocalizedStringResource`).
- `source.platform.rawValue` → replace with `source.platform.displayName`.
- `source.status.rawValue` → replace with `source.status.displayName`.

For status strings that are dynamically built with format args (e.g., `"Syncing..."`, `"Not synced yet"`, `"Synced \(count) files"`), wrap with `String(localized:)` if they're stored as `String` for use in labels; leave as `Text("...")` when rendered directly.

For plurals in `syncStatusText()` and `Label("\(n) types", ...)`:
Use the `.xcstrings` plural variant syntax. In the code site:
```swift
Label(String(localized: "\(count) type(s)"), ...)
```
Then in `Localizable.xcstrings`, give that key a plural variant with `one` and `other` forms (and `zh-Hans` variant with just `other` since Chinese has no plural form). See Task 5 for concrete catalog entries.

Short version for plural strings to introduce:
- `"\(fileCount) file(s) uploaded"`
- `"\(errorCount) error(s)"`
- `"\(typeCount) type(s)"`
- `"\(sampleCount) sample(s)"`
- `"\(resultCount) result(s)"`
- `"\(commentCount) comment(s)"`

These become keys in the catalog with pluralization. Reword slightly as needed so the English reads naturally — users won't see `(s)`, the pluralization expands to "1 file uploaded" / "5 files uploaded".

## Notes for implementer

- **SwiftUI `Text("literal")` is already localized automatically** — Xcode extracts these. The common trap is when you have `Text(viewModel.title)` where `title: String` is a plain string — that bypasses `LocalizedStringKey` resolution. Fix: type it as `LocalizedStringResource` in the model, or wrap the source with `String(localized:)` when setting.
- **`Text(_ verbatim:)`** exists specifically for non-localized strings — do NOT use it for UI text; reserve for things like hashes, version numbers, email addresses.
- **String Catalog extraction is automatic** — the `xcodebuild build` step populates empty catalogs. Don't hand-author English source; let Xcode do it.
- **For plural strings**, you must use the key format `"%lld foo(s)"` style in source and define plural variants in the catalog. `String(localized: "\(count) file(s) uploaded")` automatically maps to the key `"%lld file(s) uploaded"`.
- **Sync .xcstrings files back to Linux worktree** after every macmini build step, otherwise the committed state will diverge.
- **`LocalizedStringResource` vs `LocalizedStringKey`:** use `LocalizedStringResource` for struct fields / model types (Swift 5.9+, more flexible); use `LocalizedStringKey` implicitly via SwiftUI views (Text, Button, etc.).
- If a string literal survives in a place SwiftUI doesn't auto-localize (e.g., a `String`-typed function return used downstream in UI), wrap with `String(localized:)` so it gets extracted.
- The DataSource `description` field is kept as `LocalizedStringResource` for consistency even though not currently rendered — no wasted effort, keeps the struct uniform.
- If xcodebuild auto-extraction produces duplicate keys with different context (e.g. two `"Delete"`), it's fine; keep both entries. zh-Hans translation `删除` works in both.
- **Do NOT** add in-app language picker. Deferred.
